Why this exists

Agentic work fails in predictable ways: the agent improvises past the spec, claims tests pass without running them on a fresh dependency set, merges in-flight work while a scope question is open, picks architectural decisions silently. This plugin enforces a structural pattern that catches every one of those.

Install in 30 seconds

# In Claude Code:
/plugin install scottconverse/agentic-pipeline

# Then in any project (new or existing):
/pipeline-init

# Then:
/new-run feature my-task-slug
# (fill in the manifest the plugin scaffolds)
/run-pipeline feature 2026-05-09-my-task-slug

The plugin asks one onboarding question — what do you have? — and accepts a PRD path, a repo URL, or a description paragraph. From any of those it scaffolds .pipelines/, scripts/policy/, and a starter CLAUDE.md.

What you get

Three explicit gates

Manifest, plan, manager-decision. No autonomous-mode-by-stealth. Every gate is a one-question prompt.

Automated policy stage

Blocks the run if changes fall outside allowed_paths, contain TODO/FIXME/HACK, or modify an existing ADR.

Independent verifier

Fresh-context check against every manifest exit criterion. Cannot be talked out of NOT MET.

Manager with teeth

PROMOTE / BLOCK / REPLAN. Must cite verifier evidence verbatim. Encouragement is forbidden.

Resumable runs

Append-only run.log means re-invoking /run-pipeline picks up at the right place — every time.

Project-shaped

After init, the files are yours to edit. Add a project-specific policy check, customize a role file, define your own pipeline type.

Layers added since v0.1

Each minor version solves a specific class of failure that the previous layer didn't catch. The layers stack — most projects run all of them.

v0.2 — module-release pipeline

Six-phase release flow with Phase 0 preflight (audit the release workflow before any product code is touched) and Phase 2 local rehearsal (run the release sequence locally on fresh state before pushing the tag). Built from receipts where tags moved four times because the workflow was the discovery mechanism instead of the execution mechanism. Plus self-classification rules so the agent doesn't halt-and-ask on routine mechanical CI fixes.

Catches: execution-cascade failures — pre-existing CI bugs, tag-move dances, halt-and-ask loops.

v0.3 — Dual-AI audit-handoff

/audit-init scaffolds the three-artifact discipline for projects where one AI implements and another audits (Claude↔Codex, either direction). Implementer runs a hostile 5-lens self-audit before push; auditor runs a 10-section verification protocol; both share an in-repo doc that accumulates project-specific drift patterns over time.

Catches: drift failures — wrong endpoint, stale CHANGELOG, "Closed" without evidence, status-word abuse.

v0.4 — Judge layer (new)

Real-time action-level supervision inside the executor stage. Opt-in via .pipelines/action-classification.yaml. The orchestrator classifies every executor tool call by risk class (read_only / reversible_write / external_facing / high_risk) and spawns a context-isolated judge subagent for the dangerous ones. Four verdicts: allow, block, revise, escalate.

Catches: unauthorized actions in real time — destructive commands, external writes, force pushes, credential-touching operations, evaluated against the manifest at the action boundary instead of after the fact.

What this plugin will NOT do

  • Propose autonomous mode. Every gate is explicit.
  • Silently expand scope. The policy stage blocks changes outside allowed_paths.
  • Skip tests. CLAUDE.md hard-rule "never skip tests" is enforced as a project default.
  • Promote a run if the verifier marked any criterion NOT MET.
  • Merge in-flight work while a halt is active — even "obviously safe" cleanup PRs.

Lessons baked in

This plugin's defaults reflect failures from prior projects. Some of the more expensive ones:

  • Halts apply to ALL repo state changes. Including in-flight cleanup work. Past version of me merged a PR while the operator was still reading. Never again.
  • Tests pass locally is not evidence. CI on a fresh dependency set is. Cleanroom Docker stage is recommended for any project with non-trivial deps.
  • Manifest amendments are corrections, not expansions. If the scope grows, that's a new run, not a manifest edit.
  • The manager must cite the verifier verbatim. Summarization and encouragement are how bad runs get promoted.
  • Director decisions belong in the manifest, not in chat. The researcher surfaces them; the human picks in writing.

Stability posture

At v0.4, the structural pattern has shipped across multiple projects (CivicCast, CivicSuite, AgentSuiteLocal) and absorbed real failure receipts into its defaults. Slash-command edge cases still surface in new codebases before they surface in mine — open an issue or a discussion when something bites. Semver applies once the 0.1.x-beta tag drops; until then, CHANGELOG calls out any breaking change to slash-command arguments, manifest fields, or role-file contracts.
